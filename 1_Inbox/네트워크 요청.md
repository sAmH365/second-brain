
# 1. 웹(HTTPS) 요청의 전체 타임라인

웹사이트(예: 네이버)에 접속해서 데이터를 주고받는 전체 과정입니다. 질문하신 **3-Way Handshake, 키 교환, 세션 ID**가 모두 이 순서대로 일어납니다.

[1단계] 물리적인 길 뚫기 (TCP 3-Way Handshake)

암호화고 뭐고 간에, 먼저 두 컴퓨터 간에 통신 회선부터 연결해야 합니다.

1. **클라이언트** ➡️ **서버**: "똑똑, 연결 가능한가요?" (SYN)
2. **서버** ➡️ **클라이언트**: "네, 가능합니다. 대화 준비되셨나요?" (SYN-ACK)
3. **클라이언트** ➡️ **서버**: "네, 준비됐습니다. 연결합니다!" (ACK)

- _결과: 이제 데이터를 주고받을 통로가 열렸습니다. (이 단계를 **TCP 연결**이라고 합니다)_

[2단계] 자물쇠 채우기 (TLS 핸드셰이크 & 디피-헬먼)

길이 뚫렸으니, 이제 해커가 못 보게 암호화 통로를 만듭니다. 여기서 우리가 배웠던 **디피-헬먼(DH) 계산**이 들어갑니다.  
4. **클라이언트** ⇄ **서버**: 서로 숫자를 주고받으며 각자 컴퓨터에서 계산을 수행합니다.  
5. **결과 도출**: 계산 결과로 **[대칭키]**와 이번 연결의 이름표인 **[세션 ID]**가 동시에 만들어집니다.

- **대칭키**: 양쪽 컴퓨터 **RAM 메모리**에 꽁꽁 숨깁니다. (개발자 도구에 안 보임)
- **세션 ID**: 클라이언트와 서버가 서로 기억합니다.

[3단계] 진짜 데이터 요청 (HTTP)

이제 안전한 터널이 완성되었으니 진짜 웹페이지를 달라고 요청합니다.  
6. **클라이언트**: "메인 페이지(Index.html) 보내줘"라는 요청을 아까 만든 **[대칭키]로 암호화**해서 서버로 보냅니다.  
7. **서버**: 메모리에 있던 **[대칭키]로 복호화**해서 내용을 읽고, 웹페이지 데이터를 다시 **[대칭키]로 암호화**해서 클라이언트에게 보냅니다.  
8. **클라이언트**: 그것을 받아 화면에 띄웁니다.

[4단계] 다음 요청 보낼 때 (세션 재사용)

사용자가 웹사이트 안에서 다른 링크를 클릭했습니다. 원래대로라면 [1단계]부터 다시 해야 하지만, 리소스를 아끼기 위해 **세션 ID**를 씁니다.  
9. **클라이언트**: "나 아까 **[세션 ID: XYZ]**로 대화하던 사람인데, 다음 페이지도 보내줘!" 하고 요청합니다.  
10. **서버**: "어, 내 메모리에 [세션 ID: XYZ]랑 짝꿍이었던 **[대칭키]**가 아직 남아있네? 귀찮게 새로 계산하지 말고 **그 대칭키 그대로 쓰자!**"


# 2. SSH연결

[1단계] **TCP 3-Way Handshake**: 물리적인 통로를 뚫습니다.

[2단계]  **키 교환 (디피-헬먼)**: 서로 각자 계산해서 **[대칭키]**와 **[세션 ID]**를 먼저 만듭니다. 이제부터 모든 통신은 암호화됩니다.

[3단계]  **[차이점] 키 기반 인증 (신분증 검사)**: 암호화된 통로 안에서, 서버가 난수(Challenge)를 보냅니다. 내 컴퓨터가 내 **개인키**로 서명해서 보내면, 서버가 등록된 **공개키**로 검증합니다. 이때 위에서 만든 [세션 ID]를 난수와 함께 섞어서 서명에 사용합니다.

[4단계] **인증 성공**: 신분이 확인되면 리눅스 터미널 창이 열리고, 이후 입력하는 모든 명령어는 2번에서 만든 **대칭키**로 암호화되어 안전하게 전송됩니다.
